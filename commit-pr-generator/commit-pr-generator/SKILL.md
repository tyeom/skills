# Commit & PR Generator

git 저장소의 변경 이력을 분석해서 두 가지 결과물을 만든다:

1. **커밋 메시지** (워크트리에 변경사항이 있을 때만)
2. **Pull Request 제목과 내용** (항상)

## 절대 규칙: 워크트리를 더럽히지 말 것

이 스킬의 목적은 "분석"이다. 사용자는 커밋 직전의 민감한 상태에 있으므로, 분석 과정에서 저장소 상태가 조금이라도 바뀌면 사용자의 작업을 망칠 수 있다.

- **읽기 전용 git 명령만 사용**: `git status`, `git diff`, `git log`, `git show`, `git branch`, `git rev-parse`, `git ls-files` 등
- **금지**: `git add`, `git commit`, `git stash`, `git checkout`, `git restore`, `git clean`, 대상 저장소 내부에 파일 생성/수정/삭제
- 분석 메모가 필요하면 저장소 **바깥**(임시 폴더)에만 작성한다
- 결과물(커밋 메시지, PR 내용)은 파일로 저장하지 말고 **채팅 응답으로 출력**한다 (사용자가 파일 저장을 명시적으로 요청한 경우만 예외이며, 그 경우에도 대상 저장소 바깥에 저장)

## 분석 절차

### 1단계: 워크트리 상태 확인

```bash
git -C <repo> status --porcelain
```

- 출력이 **있으면** → 워크트리가 더러움 → 2-A단계 (변경분 분석)
- 출력이 **없으면** → 워크트리가 깨끗함 → 2-B단계 (최근 커밋 분석)

### 2-A단계: 워크트리 변경분 분석 (더러운 경우)

변경의 전체 그림을 잡기 위해 아래를 순서대로 수집한다:

```bash
git -C <repo> status --porcelain        # 변경 파일 목록 (staged/unstaged/untracked 구분)
git -C <repo> diff                      # unstaged 변경 내용
git -C <repo> diff --cached             # staged 변경 내용
```

untracked 파일은 diff에 나오지 않으므로 내용을 직접 읽는다 (읽기만 할 것). 파일이 크거나 바이너리면 파일명과 확장자로 용도를 추정한다.

diff를 읽을 때는 표면적인 줄 변경이 아니라 **의도**를 파악한다:
- 새 기능인가, 버그 수정인가, 구조 개선인가?
- 여러 파일에 걸친 변경이 하나의 논리적 변경인가, 서로 다른 변경 여러 개인가?
- 공개 인터페이스(API 시그니처, gRPC/proto, 설정 스키마)가 바뀌었는가? → 영향 범위에 중요

### 2-B단계: 최근 커밋 분석 (깨끗한 경우)

PR 대상이 될 커밋 범위를 결정한다. 우선순위:

1. **베이스 브랜치 대비 앞선 커밋**: 현재 브랜치가 main/master/develop이 아니면, 베이스 브랜치와의 차이를 분석
   ```bash
   git -C <repo> branch --show-current
   git -C <repo> log main..HEAD --oneline    # main이 없으면 master, develop 순으로 시도
   git -C <repo> diff main...HEAD --stat
   ```
2. **베이스 브랜치를 알 수 없거나 현재 브랜치가 베이스면**: 최근 커밋들을 분석 (기본 최근 5~10개, 논리적으로 한 묶음으로 보이는 범위까지)
   ```bash
   git -C <repo> log --oneline -10
   git -C <repo> show <hash> --stat          # 각 커밋의 개요
   git -C <repo> show <hash>                 # 필요하면 상세 diff
   ```

어느 범위를 선택했는지 결과 출력 시 명시한다 (예: "main 대비 3개 커밋 기준", "최근 5개 커밋 기준"). 애매하면 사용자에게 범위를 확인한다.

깨끗한 워크트리에서는 이미 커밋이 완료된 상태이므로 **커밋 메시지는 생성하지 않고 PR 내용만 생성**한다.

### 3단계: 변경 항목 정리

분석한 내용을 논리적 변경 단위로 나눈다. 파일 단위가 아니라 **의미 단위**로 묶는다. 예를 들어 컨트롤러 + 서비스 + DTO가 함께 바뀌었고 모두 "로봇 상태 조회 API 추가"라면 그것은 1개 항목이다.

각 항목에 커밋 타입을 부여한다:

| 타입 | 사용 시점 |
|---|---|
| feat | 새 기능 추가 |
| fix | 버그 수정 |
| refactor | 동작 변경 없는 구조 개선 |
| perf | 성능 개선 |
| docs | 문서만 변경 |
| test | 테스트 코드만 변경 |
| style | 포맷팅, 세미콜론 등 (동작 무관) |
| chore | 빌드 설정, 의존성, 잡무 |
| build / ci | 빌드 시스템 / CI 설정 |

타입 판단이 애매하면 보수적으로: 동작이 조금이라도 바뀌면 refactor가 아니라 feat 또는 fix다.

## 출력 형식

두 결과물을 아래 형식 그대로 출력한다. 사용자가 복사해서 바로 쓸 수 있도록 각각 코드블록에 담는다.

### 결과 1: 커밋 메시지 (워크트리가 더러운 경우만)

형식: `[{타입}] {순번}. {메시지}` — 변경 항목마다 한 줄, 순번은 1부터 순차 증가.

```
[feat] 1. 로봇 암 관절 상태 조회 gRPC 엔드포인트 추가
[feat] 2. 관절 상태 응답 DTO 및 매핑 로직 구현
[fix] 3. 토크 제한값 검증 시 음수 허용되던 버그 수정
```

항목이 1개면 한 줄만 출력한다:

```
[refactor] 1. RobotStateService의 상태 폴링 로직을 별도 모듈로 분리
```

메시지 작성 원칙:
- 한글로 작성
- "무엇을 했는지"가 아니라 "무엇이 어떻게 바뀌었는지"를 명사형 종결로 간결하게 (예: "~추가", "~수정", "~분리")
- 파일명 나열이 아니라 기능/동작 관점으로 서술

### 결과 2: Pull Request 내용 (항상)

```markdown
## PR 제목
[{대표 타입}] {전체 변경을 요약하는 한 줄}

## 요약
{이 변경이 왜 필요했고 무엇을 달성하는지 2~4문장}

## 변경사항
- {변경 항목 1 — 커밋 메시지 항목과 대응하되 한 단계 더 자세히}
- {변경 항목 2}
- ...

## 영향 범위
- {영향받는 모듈/기능과 그 이유. 예: "Proxy 모듈 — gRPC 시그니처 추가로 재생성 필요"}
- {인터페이스 변경, 설정 변경, 마이그레이션 필요 여부 등}
- {영향이 국소적이면 "영향 범위: {해당 모듈} 내부로 한정, 외부 인터페이스 변경 없음"처럼 명시}
```

대표 타입은 변경 항목 중 가장 비중이 큰 타입을 사용한다 (feat가 하나라도 핵심이면 feat).

## 예시 (전체 흐름)

**상황**: NestJS 미들웨어 저장소, 워크트리에 staged 2개 파일 + untracked 1개 파일

**출력**:

커밋 메시지:
```
[feat] 1. 양팔 로봇 충돌 감지 임계값 설정 API 추가
[chore] 2. 충돌 감지 관련 환경변수 기본값 정의
```

PR 내용:
```markdown
## PR 제목
[feat] 양팔 로봇 충돌 감지 임계값 설정 기능 추가

## 요약
기존에는 충돌 감지 임계값이 코드에 하드코딩되어 있어 현장별 튜닝이 불가능했습니다.
이 PR은 임계값을 런타임에 설정할 수 있는 API를 추가하고, 환경변수로 기본값을 관리하도록 변경합니다.

## 변경사항
- CollisionController에 PUT /collision/threshold 엔드포인트 추가 (DTO 검증 포함)
- CollisionService에 임계값 동적 반영 로직 구현
- .env.example에 COLLISION_THRESHOLD_DEFAULT 항목 추가

## 영향 범위
- 충돌 감지 모듈 내부로 한정, 기존 API 시그니처 변경 없음
- 배포 시 환경변수 COLLISION_THRESHOLD_DEFAULT 설정 필요 (미설정 시 기존 하드코딩 값과 동일한 기본값 사용)
```

## 마무리 체크

출력 전에 스스로 확인한다:
- [ ] 분석 과정에서 워크트리를 변경하는 명령을 하나도 실행하지 않았는가?
- [ ] 커밋 메시지 순번이 1부터 빠짐없이 이어지는가?
- [ ] 각 항목의 타입이 실제 변경 성격과 일치하는가?
- [ ] PR 영향 범위에 인터페이스/설정 변경 여부가 명시되어 있는가?
